# P1.3b — Cron do pipeline de enriquecimento geo

## Objetivo

Executar o worker `geo-enrichment-worker` a cada 5 minutos para processar apoiadores com `geo_pending` ou `geo_enrichment_failed`, sem bloquear a landing e sem HTTP dentro de PL/pgSQL.

## Configuração (manual)

No **Supabase Dashboard**:

1. **Edge Functions** → `geo-enrichment-worker` → **Deploy** (ver checklist de deploy abaixo).
2. **Integrations** → **Cron** (Scheduled Functions) → **Create job**:
   - **Schedule:** `*/5 * * * *` (a cada 5 minutos)
   - **Function:** `geo-enrichment-worker`
   - **HTTP method:** `POST`
   - **Headers:** usar a **service role key** em `Authorization: Bearer <SERVICE_ROLE_KEY>` e `apikey: <SERVICE_ROLE_KEY>` (mesmo padrão de invocação server-side).

> **Não usar** `pg_cron`, `pg_net` nem chamadas HTTP em funções PL/pgSQL.

## Fluxo

1. `process_pending_geo_enrichment(100)` — cache hits via `get_postal_code_cache` + `apply_supporter_geo_from_cep`.
2. `claim_geo_enrichment_batch(50)` — cache misses com `FOR UPDATE SKIP LOCKED`.
3. Para cada item: `resolve-postal-code` → `apply_supporter_geo_from_cep` ou `fail_geo_enrichment`.
4. Logs em `geo_enrichment_logs`.

## Retenção de logs

Recomendado: cleanup de registros com `created_at < now() - interval '14 days'` (job SQL ou manutenção periódica). Documentado; não automatizado nesta fase.

## Validação (P1.3b — checklist)

```sql
-- 1) CEP válido → enrich (após worker ou cache hit SQL)
SELECT id, cep, geo_pending, geo_enriched_at, geo_source, city, state_uf
FROM supporters WHERE cep IS NOT NULL ORDER BY created_at DESC LIMIT 10;

-- 2) Fila pendente
SELECT COUNT(*) FROM supporters
WHERE geo_pending AND cep IS NOT NULL;

-- 3) manual > viacep (prioridade)
SELECT public.geo_source_priority('manual'), public.geo_source_priority('viacep');

-- 4) Métricas por tenant
SELECT * FROM geo_enrichment_metrics_v WHERE tenant_id = '<TENANT_UUID>';

-- 5) Logs recentes
SELECT * FROM geo_enrichment_logs ORDER BY created_at DESC LIMIT 20;

-- 6) Falhas
SELECT id, cep, geo_enrichment_failed, geo_last_error, geo_enrichment_attempts
FROM supporters WHERE geo_enrichment_failed LIMIT 20;

-- 7) RPC SQL (service role) — cache hits sem HTTP
SELECT public.process_pending_geo_enrichment(10);

-- 8) Telefone normalizado (regressão P1.3a)
SELECT phone FROM supporters WHERE phone ~ '[^0-9]' LIMIT 5;
```

Invocar worker manualmente (service role):

```bash
curl -X POST "$SUPABASE_URL/functions/v1/geo-enrichment-worker" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY" \
  -H "apikey: $SUPABASE_SERVICE_ROLE_KEY"
```

## Deploy das functions

```bash
supabase functions deploy resolve-postal-code
supabase functions deploy geo-enrichment-worker
```

Aplicar migration:

```bash
supabase db push
# ou repair + push conforme histórico do projeto
```
