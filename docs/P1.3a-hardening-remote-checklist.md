## 2. RPCs e objetos P1.3a

```sql
-- Funções críticas
SELECT proname, pg_get_function_identity_arguments(p.oid) AS args
FROM pg_proc p
JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'public'
  AND proname IN (
    'apply_supporter_geo_from_cep',
    'register_supporter_from_landing',
    'get_tenant_operational_dashboard',
    'normalize_cep',
    'normalize_supporter_phone'
  )
ORDER BY proname, args;

-- Tabela cache
SELECT EXISTS (
  SELECT 1 FROM information_schema.tables
  WHERE table_schema = 'public' AND table_name = 'postal_code_cache'
) AS postal_code_cache_exists;

-- Colunas geo em supporters
SELECT column_name
FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'supporters'
  AND column_name IN ('cep', 'geo_pending', 'state_uf', 'ibge_city_code')
ORDER BY 1;
```

**Confirmado no remoto (auditoria):**

| Item | Status |
|------|--------|
| `postal_code_cache` | ✅ |
| `supporters.cep`, `geo_pending`, `state_uf` | ✅ |
| `apply_supporter_geo_from_cep` | ✅ |
| `register_supporter_from_landing(..., text, text)` com `p_email` + `p_cep` | ✅ overload 10 params |
| `get_tenant_operational_dashboard` contém `territory_key` no corpo | ✅ (`prosrc LIKE '%territory_key%'`) |

## 3. Dashboard — territory_key em produção

```sql
SELECT prosrc LIKE '%supporter_territory_key_row%' AS uses_normalized_key,
       prosrc LIKE '%trim(s.neighborhood)%' AS still_uses_trim
FROM pg_proc p
JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'public' AND p.proname = 'get_tenant_operational_dashboard';
```

**Esperado:** `uses_normalized_key = true`, `still_uses_trim = false`.

Teste funcional (substituir UUID do tenant):

```sql
SELECT jsonb_path_exists(
  public.get_tenant_operational_dashboard('TENANT_UUID'::uuid)::jsonb,
  '$.insights.critical_territories[0].territory_key'
) AS has_territory_key_in_insights;
```

> O payload exato depende de `build_dashboard_insights`; territórios brutos incluem `territory_key` no array agregado antes dos insights.
